Web Accessibility Evaluation for Fairdata Services

Overview

The Problem

Web accessibility has not been emphasized during the design and development of digital products within CSC. While the EUâ€™s Accessibility Act and the Finnish Digital Services Act oblige public sectors to comply with the accessibility requirement. The Fairdata service is offered by the Ministry of Education and provided by CSC-IT Center for Science. Therefore, Fairdata services need to reach WCAG 2.1 AA level.

My Role

Design for Web Accessibility, Accessibility Evaluation, Screen Reader Testing

Results

Identified all accessibility issues and fixed these issues accordingly so that the web service meets WCAG 2.1 AA level.

My Process

Define the scope -> Find tools -> Preliminary evaluation with automated testing tools -> Comprehensive evaluation with manual testing -> Sort out and report findings -> Fix text alternatives issues and discuss the rest with developers

Tool Used

Adobe XD, WAVE, ARC Toolkit, NVDA, Voiceover

Timeline

April to August 2020

Accessibility Issues

The major accessibility issues are alternative text, contrast ratio, and keyboard-only navigation. These issues can be easily resolved by adding descriptive alternative text, changing colour combinations, and using ARIA labels.

Text Alternatives

All non-text content should have a text alternative that conveys the equivalent meaning and purposes. Text alternatives are one of the main approaches to make information accessible since text can be presented in any sensory modality.

  1. All images do not have proper text alternatives
  2. Videos do not have a title attribute
  3. Videos do not have Finnish captions:

    People with cognitive disabilities have difficulties in comprehending texts and static images. They rely on the video to understand the contents. Thus, it is also crucial to provide captions in Finnish.

  4. Tables on the website do not have table headers
Contrast Ratio

In web accessibility, the contrast ratio is to check whether the contrast between the text and background is sufficient. There are diverse colour vision deficiencies making it impossible to specify a universal colour pair based on the quantitative data. For different types of disabilities, the requirement of contrast ratio is opposite, especially for people with dyslexia, high luminance is not readable. In order to overcome this problem, it is recommended to provide a mechanism to adjust the foreground and background colors of the content. Of course, individuals with colour vision deficiencies also rely on some assistive technologies to achieve an appropriate contrast ratio.

The minimal contrast ratio legally required by WCAG 2.1 is 1:4.5. The goal is to provide sufficient contrast between the text and the background so that people with low vision can read the text, but does not include people who use contrast-enhancing assistive technology.

As the image on the right, the main contrast error happens on the side navigation. Moreover, the colour #007FAD and #E5F2F7 are two primary colours on the website. Another colour insufficiency is on the text and button background.

Keyboard-only Navigation

All elements, including links, buttons, and media player controls, should be accessible using the Tab key. Meanwhile, the keyboard should be able to activate actions, options, visible changes, and other functionalities. All these functionalities should reveal a visible focus state and respond to the common key presses.

Keyboard-only navigation is the most severe accessibility issue on the Fairdata site. The tab order is poorly structured on the home page and FAQ page. There are keyboard traps and lacking aria-label.

Note, terrible tab order overwhelms people with disabilities and would eventually contribute to a high drop off rate.

Compatibility

Screen reader testing is applied to inspect the compatibility with different browsers and to detect unexpected practical issues. NVDA is the most popular screen readers among people with disabilities, and the use of Voiceover is steadily increasing. On the other side, Chrome is the most used browser, and Firefox is the second. So, I use NVDA and Voiceover to test the compatibility with Chrome and Firefox. Each screen reader browses headings, links, tables, images, and videos in two browsers. Then read through the entire web page to check that the content is well-organized and logical.

The website is compatible with both screen readers to some extent, but the performance of the two browsers is different. Compared with Firefox, Chrome is a better choice for both Voiceover and NVDA users, especially for the Voiceover users. Firefox is still working on the compatibility with Voiceover.

Scope

The evaluation is carried out on all web contents of the public website, which is located at http://www.fairdata.fi. This website was published on 14 June 2018 so that it should be accessible before 23rd September 2020.

The minimal conformance target is AA level criteria of the WCAG 2.1 guidelines. The only exception is 1.2.4 Subtitling (for live broadcasts), which do not have to meet the criteria. Thus, there are a total of 49 criteria required by Finnish law.

The natural languages of the website are English and Finnish. The content of the Finnish and English versions of the website are slightly different, but the basic structure is the same. In this study, the evaluation is only conducted on the English version. The Finnish version can be improved based on the findings in the English version.

The types of website content are quite fundamental and visual-based. Therefore, the study aims at improving accessibility for users with visual-related disabilities, auditory disabilities, and cognitive disabilities.

The website should support Voiceover and NVDA on Chrome and Firefox, which are the most popular combinations in each operating system. Since the website is primarily designed for desktop access, mobile accessibility is not considered in this evaluation, although the website supports mobile access. Also, other assistive technologies are not evaluated in this study.

Reflection

Web accessibility can't be easily achieved by complying with the WCAG 2.1. It depends highly on the expertise of the evaluator to ensure quality. However, the most experienced evaluator can't replace the real user. The experience from the real user is significant.

We have fixed these accessibility issues together with developers.

Future work should involve target users in the web accessibility evaluation so that more real-world accessibility issues can be discovered. With feedback from real users, the project team can concurrently deepen the understanding of people with disabilities. On the other side, accessibility evaluation, especially manual testing, should be an iterated process incorporated in the regular maintenance to ensure web accessibility.
